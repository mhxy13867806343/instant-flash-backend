# 移动端账号注销设计

## 目标

为用户端账号提供申请注销和撤销注销能力。申请成功后进入 60 天等待期；等待期内账号仍可登录并撤销申请，截止时间到达后账号永久注销。申请和撤销仅允许移动端账号操作。

## 接口

### 申请注销

- 方法与路径：`POST /api/user/deactivation`
- 鉴权：Bearer Token
- 请求体：`{"reason": "注销原因"}`
- 原因处理：去除首尾空白后，长度必须为 10–500 个 Unicode 字符。
- 移动端校验：当前账号的 `clientType` 必须为 `android`、`ios`、`harmonyos`、`miniprogram` 或 `h5`。
- 成功后：
  - `deactivationStatus` 设为 `pending`；
  - `deactivationReason` 保存规范化后的原因；
  - `deactivationApplyTime` 设为当前 UTC 时间；
  - `deactivationEndTime` 设为申请时间加 60 天；
  - 返回申请时间、截止时间和注销状态。
- 已处于 `pending` 时重复申请返回 400。

### 取消注销

- 方法与路径：`POST /api/user/deactivation/cancel`
- 鉴权：Bearer Token
- 移动端校验与申请接口相同。
- 仅当状态为 `pending` 且当前时间严格早于 `deactivationEndTime` 时允许取消。
- 成功后清空注销状态、原因、申请时间和截止时间，并保持 `isActive=true`。
- 未申请注销时返回 400。
- 当前时间等于或晚于截止时间时，先永久注销账号，再拒绝取消。

## 生命周期与访问控制

- 等待期内账号保持启用，可正常登录和访问用户端接口，以便撤销申请。
- 当系统在鉴权或登录时发现 `pending` 账号已到截止时间，将其状态更新为 `deactivated`、将 `isActive` 设为 `false`，并拒绝访问或登录。
- 登录流程必须持久化到期状态，不能只修改当前 ORM 对象后因异常回滚而丢失。
- `deactivated` 账号不能重新登录，也不能调用取消接口。
- 本次不增加定时任务；到期状态在账号下一次登录或鉴权时惰性落库。

## 数据与返回

沿用当前半成品迁移中的四个用户字段：

- `deactivation_status`
- `deactivation_reason`
- `deactivation_apply_time`
- `deactivation_end_time`

个人资料序列化需真实返回对应的 camelCase 字段，不能始终返回默认空值。

## 错误处理

- 非移动端账号：400，提示仅支持移动端操作。
- 原因不足 10 字或超过 500 字：422，沿用全局请求校验响应。
- 重复申请：400。
- 未申请时取消：400。
- 已过 60 天：账号永久注销；鉴权接口返回 401，登录接口返回 403。

## 测试

测试覆盖：

- 五类允许的移动端类型及非移动端拒绝行为；
- 去除首尾空白后的 10 字与 500 字边界；
- 成功申请时四个注销字段及精确 60 天截止时间；
- 重复申请被拒绝；
- 等待期内账号可登录、可取消，取消后字段清空；
- 截止时间到达后不可取消、不可鉴权、不可登录，并持久化永久注销状态；
- 个人资料正确透出注销状态和时间。

## 非目标

- 不物理删除用户及其内容。
- 不新增后台管理端注销入口。
- 不新增定时清理任务或通知系统。
- 不允许永久注销后恢复账号。
